# Notifications

## Summary

The Notifications system keeps you informed about important activity across the Getswyft platform. Notifications are delivered in-app and can be pushed to mobile devices via Firebase Cloud Messaging (FCM). You can view your notification history, mark items as read, register devices for push alerts, and send test notifications to verify your setup.

## Who Can Use This

- **All authenticated users** can view and manage their own notifications. No specific permission beyond authentication is required.
- Push notifications require a registered device with a valid FCM token.

## What It Does

The notification system provides:

- **In-app notifications** — A chronological list of alerts about activity relevant to you, accessible through the API and the website header.
- **Push notifications** — Real-time alerts delivered to registered mobile or desktop devices via Firebase Cloud Messaging.
- **Device management** — Register and unregister devices to control where push notifications are sent.
- **Read tracking** — Mark notifications as read to keep your notification list organized.
- **Test notifications** — Send a test push notification to verify device registration and delivery.

## Key Functions / Actions Available

| Action | API Endpoint | Method | Description |
|---|---|---|---|
| List notifications | `GET /v1/notifications` | GET | Retrieve your notifications. Supports a `limit` parameter (20–100). |
| Mark as read | `POST /v1/notifications/:id/read` | POST | Mark a specific notification as read. |
| Register device | `POST /v1/notifications/devices` | POST | Register a device token for push notifications. |
| Remove device | `DELETE /v1/notifications/devices` | DELETE | Unregister a device to stop receiving push notifications on it. |
| Send test notification | `POST /v1/notifications/test` | POST | Send a test push notification to your registered devices. |

## Step-by-Step How to Use It

### Viewing Notifications

1. In the website, click the **notification bell icon** in the header to see recent alerts.
2. Via the API, send a `GET` request to `/v1/notifications` with an optional `limit` query parameter (between 20 and 100) to control how many notifications are returned.
3. Each notification includes:
   - `type` — The category of the notification (e.g., conversation assignment, new message).
   - `title` — A brief headline.
   - `body` — Detailed notification text.
   - `payload` — A JSON object with additional context (e.g., conversation ID, message ID).
   - `readAt` — Timestamp when the notification was read, or `null` if unread.

### Marking a Notification as Read

1. Send a `POST` request to `/v1/notifications/:id/read`, where `:id` is the notification ID.
2. The `readAt` field is updated to the current timestamp.
3. The notification no longer appears as unread in the website header badge.

### Registering a Device for Push Notifications

1. Obtain an FCM device token from your mobile or desktop application.
2. Send a `POST` request to `/v1/notifications/devices` with:
   - `token` (string, required) — The FCM device token.
   - `deviceLabel` (string, optional) — A human-readable label for the device (e.g., "John's iPhone").
3. The device is registered and will begin receiving push notifications.

### Removing a Device

1. Send a `DELETE` request to `/v1/notifications/devices` with the device token to unregister.
2. The device will no longer receive push notifications.

### Sending a Test Notification

1. Send a `POST` request to `/v1/notifications/test`.
2. A test push notification is delivered to all of your registered devices.
3. Use this to verify that device registration and FCM delivery are working correctly.

## System Behavior / What Users Should Expect

- **Notification triggers:** Notifications are automatically generated by the following events:
  - A conversation is assigned to you or updated.
  - A new message is received in a conversation you are part of.
  - An invitation is sent, accepted, or otherwise updated.
- **Push delivery:** Push notifications are sent via Firebase Cloud Messaging. Delivery depends on the device being online and having a valid, registered FCM token.
- **In-app alerts:** The website displays a notification indicator (bell icon) in the header. Unread notifications are highlighted with a badge count.
- **Read status:** Notifications start as unread (`readAt` is `null`). Once marked as read, they are no longer counted in the unread badge.
- **Retention:** Notifications are stored and retrievable through the API. The `limit` parameter controls how many are returned per request (minimum 20, maximum 100).

## Permissions Required

| Requirement | Details |
|---|---|
| Authentication | A valid session or API token is required. |
| No additional permissions | All authenticated users can access their own notifications. |

Notifications are scoped to the authenticated user. You cannot view or manage another user's notifications.

## Common Issues

| Issue | Cause | Resolution |
|---|---|---|
| Not receiving push notifications | Device token is not registered or is expired. | Re-register the device using `POST /v1/notifications/devices` with a fresh FCM token. |
| Push notification delayed | Network connectivity or FCM service latency. | Verify the device has an active internet connection. Check FCM service status. |
| Notification badge not updating | Notification has not been marked as read. | Use `POST /v1/notifications/:id/read` to mark notifications as read. |
| Test notification not received | No devices registered for the user. | Register at least one device before sending a test notification. |
| Too few notifications returned | Default limit may be lower than expected. | Specify a higher `limit` parameter (up to 100) in the `GET /v1/notifications` request. |

## Support Notes / Troubleshooting

- If push notifications are not arriving, start by sending a test notification via `POST /v1/notifications/test`. If the test is not received, the issue is likely with device registration or the FCM token.
- FCM tokens can expire or become invalid if the app is reinstalled or the device is reset. Re-register the device when this occurs.
- The notification bell in the website header reflects the current unread count. If the count seems incorrect, refresh the page or check the API response for accurate `readAt` values.
- For integration or automation purposes, use the API endpoints directly. The notification list endpoint supports pagination through the `limit` parameter.
